# Database Scanning

Direct database row scanning into Atoms without intermediate allocations.

## Overview

Atom provides a `Scanner` type that efficiently scans database rows directly into `*Atom` instances. This bypasses the typical pattern of:

```
rows.Scan() -> struct -> Atomize() -> *Atom
```

Instead, Scanner goes directly:

```
rows.Scan() -> *Atom
```

## The `db` Struct Tag

Scanner uses the `db` struct tag to map database columns to struct fields:

```go
type User struct {
    ID    int64  `db:"id"`
    Name  string `db:"name"`
    Email string `db:"email"`
}
```

Fields without `db` tags are excluded from scanning.

### Tag Rules

1. **Scalars and pointers** - Can have `db` tags
2. **Nested structs** - Must NOT have `db` tags (their fields can)
3. **Slices** - Cannot have `db` tags (not supported for scanning)

```go
// Correct
type User struct {
    ID      int64   `db:"id"`       // OK - scalar
    Name    *string `db:"name"`     // OK - pointer (nullable)
    Address Address                 // OK - no tag on struct
}

type Address struct {
    Street string `db:"street"`     // OK - nested field has tag
    City   string `db:"city"`
}

// Errors at registration
type Bad struct {
    Address Address `db:"address"`  // ERROR: struct field cannot have db tag
    Tags    []string `db:"tags"`    // ERROR: slice field cannot have db tag
}
```

## Basic Usage

### Setup

```go
// Register the type - this also builds the Scanner
atomizer, err := atom.Use[User]()
if err != nil {
    log.Fatal(err)
}

// Retrieve the scanner
scanner, ok := atom.ScannerFor(atomizer.Spec())
if !ok {
    log.Fatal("scanner not available")
}
```

### Scanning a Single Row

```go
rows, err := db.Query("SELECT id, name, email FROM users WHERE id = $1", userID)
if err != nil {
    return err
}
defer rows.Close()

if rows.Next() {
    atom, err := scanner.Scan(rows)
    if err != nil {
        return err
    }

    fmt.Println(atom.Ints["ID"])      // 123
    fmt.Println(atom.Strings["Name"]) // "Alice"
}
```

### Scanning Multiple Rows

```go
rows, err := db.Query("SELECT id, name, email FROM users")
if err != nil {
    return err
}
defer rows.Close()

atoms, err := scanner.ScanAll(rows, rows.Next)
if err != nil {
    return err
}

for _, atom := range atoms {
    fmt.Printf("User: %s\n", atom.Strings["Name"])
}
```

## Nullable Columns

Pointer fields map to nullable columns:

```go
type User struct {
    ID       int64   `db:"id"`
    Name     string  `db:"name"`      // NOT NULL
    Nickname *string `db:"nickname"`  // NULL allowed
}
```

When scanned:
- Non-null value: `atom.StringPtrs["Nickname"]` points to the value
- NULL value: `atom.StringPtrs["Nickname"]` is `nil`

```go
atom, _ := scanner.Scan(rows)

if atom.StringPtrs["Nickname"] != nil {
    fmt.Println(*atom.StringPtrs["Nickname"])
} else {
    fmt.Println("No nickname set")
}
```

## Nested Structs

Scanner flattens nested structs into the same row:

```go
type User struct {
    ID      int64   `db:"id"`
    Name    string  `db:"name"`
    Address Address // No db tag on struct itself
}

type Address struct {
    Street string `db:"street"`
    City   string `db:"city"`
}
```

Query:
```sql
SELECT id, name, street, city FROM users
```

Result:
```go
atom.Ints["ID"]                        // 123
atom.Strings["Name"]                   // "Alice"
atom.Nested["Address"].Strings["Street"] // "123 Main St"
atom.Nested["Address"].Strings["City"]   // "NYC"
```

### Deeply Nested

```go
type Order struct {
    ID       int64    `db:"id"`
    Customer Customer
}

type Customer struct {
    Name    string  `db:"customer_name"`
    Address Address
}

type Address struct {
    City string `db:"customer_city"`
}
```

All fields flatten to the same row:
```sql
SELECT id, customer_name, customer_city FROM orders
```

## ColScanner Interface

Scanner works with any type implementing `ColScanner`:

```go
type ColScanner interface {
    Columns() ([]string, error)
    Scan(dest ...any) error
    Err() error
}
```

This is satisfied by:
- `*sql.Rows` (standard library)
- `*sqlx.Rows` (sqlx package)
- Custom implementations

## Error Handling

### Registration Errors

Errors during `Use[T]()` if db tags are misconfigured:

```go
// Struct field with db tag
type Bad struct {
    Address Address `db:"addr"` // Error: struct fields cannot have db tags
}

atomizer, err := atom.Use[Bad]()
// err: "field Address: struct fields cannot have db tags"
```

### Column Collision

Two fields mapping to the same column name:

```go
type Bad struct {
    UserName string `db:"name"`
    FullName string `db:"name"` // Same column!
}

atomizer, err := atom.Use[Bad]()
// err: "column \"name\" maps to multiple fields: UserName and FullName"
```

### Scan Errors

Runtime errors during scanning:

```go
atom, err := scanner.Scan(rows)
if err != nil {
    // "getting columns: ..." - failed to get column names
    // "scanning row: ..." - failed to scan values
}
```

### Unknown Columns

Columns in the result set without matching `db` tags are silently ignored:

```go
// Query returns: id, name, unknown_column
// Only id and name are mapped; unknown_column is discarded
```

## Converting to Structs

After scanning, convert atoms back to structs:

```go
atoms, _ := scanner.ScanAll(rows, rows.Next)

users := make([]*User, len(atoms))
for i, atom := range atoms {
    user, err := atomizer.Deatomize(atom)
    if err != nil {
        return err
    }
    users[i] = user
}
```

## Performance

Scanner is optimized for:

1. **No intermediate allocations** - Scans directly into typed destinations
2. **Destination reuse** - `ScanAll` reuses scan buffers across rows
3. **Pre-built plans** - Column mapping computed once at registration

For high-throughput scenarios, consider implementing `Atomizable`/`Deatomizable` interfaces to also bypass reflection during deatomization.

## Limitations

### Nullable uint64

Go's `database/sql` package lacks `sql.NullUint64`. Scanner uses `sql.NullInt64` for nullable unsigned integers:

```go
type Counter struct {
    Value *uint64 `db:"value"` // Scanned via NullInt64
}
```

**Limitation:** Values exceeding `MaxInt64` (9223372036854775807) will overflow when stored as negative int64 and convert incorrectly.

For large unsigned values, use non-nullable `uint64` or handle at the application level.

### Slices Not Supported

Slice fields cannot be scanned from database rows:

```go
type User struct {
    Tags []string `db:"tags"` // ERROR at registration
}
```

Use separate queries or JSON columns with custom handling.

### Maps Not Supported

Map fields are not supported by atom:

```go
type User struct {
    Metadata map[string]string // ERROR: unsupported type
}
```

## Best Practices

### Match Column Names

```go
// Good - column names match db tags
type User struct {
    ID   int64  `db:"id"`
    Name string `db:"name"`
}
// SELECT id, name FROM users

// Confusing - different conventions
type User struct {
    UserID   int64  `db:"user_id"`
    FullName string `db:"full_name"`
}
```

### Use Pointers for Nullable

```go
// Good - explicit nullability
type User struct {
    Name     string  `db:"name"`      // Required
    Nickname *string `db:"nickname"`  // Optional
}

// Risky - zero value ambiguous
type User struct {
    Name     string `db:"name"`
    Nickname string `db:"nickname"` // "" means null or empty?
}
```

### Keep Nesting Shallow

```go
// Good - one level
type Order struct {
    ID       int64   `db:"id"`
    Customer Customer
}

// Complex - multiple levels harder to query
type Order struct {
    ID       int64 `db:"id"`
    Customer struct {
        Info struct {
            Address struct {
                City string `db:"city"`
            }
        }
    }
}
```
